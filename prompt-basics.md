## 第1部：基礎理論

### 1.1 プロンプトってなんだろう？

プロンプト（Prompt）とは、AI言語モデルに「こうしてほしい」と伝えるためのテキスト入力のことです。私たち人間が、自然な言葉でAIに指示を出すための窓口だと考えてください。

そして、ここが大切なポイントなのですが、**プロンプトの質が、AIの回答の質を決める**んです。

#### なぜプロンプトが重要なの？

AIの大規模言語モデル（LLMと呼ばれます）は、あなたが入力したテキストを読んで、「次に続くのはこんな内容だろう」と予測しながら回答を作っていきます。つまり：

- **入力の質 = 出力の質**: 曖昧な質問には曖昧な答えが返ってきますし、明確な質問には明確な答えが返ってきます
- **文脈の力**: AIはプロンプトの中にある背景情報や状況を理解して、それに合わせた回答を作ります
- **例から学ぶ**: プロンプトの中に具体例を示してあげると、AIはそのパターンを学んで、同じようなスタイルで答えてくれます（これを Few-shot Learning と言います）

#### プロンプトエンジニアリングって何？

「良いプロンプトを作って、どんどん改善していくこと」を**プロンプトエンジニアリング**と呼びます。難しそうな名前ですが、やることはシンプルです：

- プロンプトをわかりやすく整理する
- タスクに合った書き方を選ぶ
- 何度か試しながら改善する
- 期待した答えが返ってくるまで調整する

最初から完璧なプロンプトを書く必要はありません。少しずつ試しながら上達していけば大丈夫です。

### 1.2 AIはどうやって動いているの？（かんたん版）

良いプロンプトを書くために、AIがどう動いているのか、ざっくり理解しておきましょう。

#### トークンって何？AIはどうやって文章を作るの？

**1. トークン化**

AIは、あなたが入力した文章を「トークン」という小さな単位に切り分けて読んでいます。

- 日本語の場合、1文字だったり、いくつかの文字のかたまりが1トークンになります
- 例：「こんにちは」→ [「こん」、「にち」、「は」] のように分解されます

**2. 確率で次の言葉を予測**

AIは、学習したデータのパターンから「次に来る言葉は何が一番自然か？」を確率で計算して選んでいます。

- 「今日の天気は」の次には「晴れ」「雨」「曇り」などが来やすい、というパターンを学習しています
- 設定次第で、正確さ重視にしたり、ちょっと創造的な回答にしたりできます

**3. コンテキストウィンドウ（一度に覚えられる量）**

AIが一度に読める文章の量には限界があります（コンテキストウィンドウと言います）。

- Claude 3の場合、だいたい20万トークン分くらい覚えられます（これはかなり多い方です）
- 長いプロンプトを書くときは、大事な情報を最初と最後に置くと効果的です

### 1.3 プロンプトの基本レシピ

効果的なプロンプトには、6つの材料があります。全部使う必要はありませんが、知っておくと便利です：

```
[役割] + [背景情報] + [やってほしいこと] + [ルール] + [形式] + [例]
```

それぞれ詳しく見ていきましょう。

#### 材料1: 役割定義（Role）
「AIにどんなキャラクターになってほしいか」を伝えます。

**こんな効果があります：**
- 答え方のトーンや専門性が変わります
- その分野の知識をうまく引き出せます

**例えば：**
```
あなたは10年以上の経験を持つベテランエンジニアです。
システム設計とパフォーマンスの改善が得意です。
```

このように役割を与えると、AIは「経験豊富なエンジニア」として答えてくれるようになります。

#### 材料2: コンテキスト（背景情報）
「今どんな状況なのか」「誰に向けて説明するのか」を伝えます。

**伝えるべき情報：**
- 誰に向けた内容なのか（技術レベル、年齢、経験）
- どんな環境で使うのか
- 今どんな問題や状況があるのか

**例えば：**
```
# 背景情報
- 読む人: プログラミング初心者（勉強して3ヶ月）
- 使う環境: Python 3.11、Windows 11
- 今の状況: if文やfor文は分かるけど、関数はまだ勉強してない
```

#### 材料3: タスク（やってほしいこと）
「具体的に何をしてほしいのか」をはっきり伝えます。

**コツ：**
- 「〜してください」という動詞で始めましょう
- あいまいな表現を避けましょう
- 1つのプロンプトには1つのメインタスクを

**例えば：**
```
次のPythonコードを読みやすく書き直して、
動作も速くなるように改善してください。
```

#### 材料4: 制約条件（ルール）
「守ってほしいこと」「やってはいけないこと」を伝えます。

**よくある制約：**
- 文字数の制限
- 使っていい技術やライブラリ
- やってはいけないこと
- 話し方のスタイル

**例えば：**
```
ルール:
- 500文字以内で説明してください
- 難しい専門用語は使わず、たとえ話で説明してください
- コードの例は必要ありません
```

#### 材料5: 出力形式（どんな形で返してほしいか）
「どんなフォーマットで答えてほしいか」を指定します。

**形式の例：**
- 表やJSON、YAMLなどのデータ形式
- 見出しや箇条書きを使ったマークダウン
- あなたが用意したテンプレート

**例えば：**
```
次の形式で答えてください：

## 概要
[2〜3文で簡単に説明]

## 詳しい説明
### 良いところ
- [良いところ1]
- [良いところ2]
- [良いところ3]

### 気をつけるべきところ
- [注意点1]
- [注意点2]

## コードの例
[実際に動くコードを書いてください]
```

#### 材料6: 例示（お手本を見せる）
「こんな感じで答えてね」という具体例を見せます。

**例を示すとこんなに便利：**
- AIは例からパターンを学んでくれます
- 例なし（ゼロショット）よりも、例あり（フューショット）の方が正確になります

**例えば：**
```
次の例と同じように処理してください：

入力: "明日は雨が降るでしょう"
出力: {"text": "明日は雨が降るでしょう", "sentiment": "中立", "entities": ["明日", "雨"]}

入力: "この商品は最高です！"
出力: {"text": "この商品は最高です！", "sentiment": "ポジティブ", "entities": ["商品"]}

では、こちらも同じように処理してください：
入力: "対応が遅すぎて困っています"
```

このように例を2〜3個見せると、AIは「ああ、こういう形式で返せばいいんだな」と理解してくれます。

## 第2部：プロンプト設計の原則

ここからは、「良いプロンプト」を書くための6つの大切な考え方を学んでいきます。最初から全部完璧にできなくても大丈夫。少しずつ意識していけば、自然と身についていきますよ。

### 2.1 原則その1：明確性（はっきり伝えよう）

**ポイント：** あいまいな表現を避けて、何をしてほしいのか正確に伝えましょう

#### 明確に伝えるコツ

**コツ1: 5W1Hを意識する**

新聞記事を書くときのように、次のことを意識すると伝わりやすくなります：

- **What（何を）**: どんなタスクをしてほしいのか
- **Why（なぜ）**: 何のためにやるのか
- **Who（誰に）**: 誰に向けた内容なのか
- **When（いつ）**: 時間的な制約はあるか
- **Where（どこで）**: どんな範囲・環境で使うのか
- **How（どうやって）**: どんな方法や形式がいいのか

**コツ2: あいまいな言葉に注意**

よく使ってしまう「あいまいな表現」とその改善例を見てみましょう：

| あいまいな表現 | もっと明確な表現 |
|----------|----------|
| "詳しく説明して" | "500文字くらいで、原因・影響・対策の3つを説明して" |
| "良いコードを書いて" | "読みやすくて、処理速度がO(n)以下になるコードを書いて" |
| "簡単に説明して" | "中学生でもわかるように、難しい言葉を使わずに説明して" |
| "最近のデータ" | "2024年以降のデータ" |

**コツ3: 良い例と悪い例を比べてみよう**

❌ **良くない例：**
```
プログラミングについて教えて
```

これだと何が問題でしょうか？
- 範囲が広すぎます（言語の話？プログラミングの歴史？基本の考え方？）
- どれくらい詳しく説明すればいいか分かりません
- 初心者向けか、経験者向けかも分かりません

✅ **改善した例：**
```
Python言語における関数の定義方法について、
プログラミング初心者（勉強して1ヶ月）向けに説明してください。

次の点を含めてください：
1. 基本的な書き方（def文の使い方）
2. 引数と戻り値って何？
3. 簡単な実例（2〜3個）

文字数: 600〜800文字くらいでお願いします
```

こうすると、AIは「ああ、初心者向けに、関数の書き方を具体例つきで説明すればいいんだな」と理解できます。

### 2.2 原則その2：具体性（具体的に言おう）

**ポイント：** ふわっとした抽象的な言い方じゃなく、具体的な情報を伝えましょう

#### 具体的に伝えるコツ

**コツ1: 実際のデータを見せる**

❌ **良くない例：**
```
このデータを分析して
```

データがどんなものか分からないと、AIも困ってしまいます。

✅ **改善した例：**
```
次の売上データを分析して、傾向とおかしな値を見つけてください：

2024年1月: 150万円
2024年2月: 148万円
2024年3月: 165万円
2024年4月: 142万円
2024年5月: 180万円
2024年6月: 95万円  ← 他の月と比べて異常に低い
```

実際のデータを示すことで、AIは「ああ、6月だけ急に落ちてるな」と気づけます。

**コツ2: 「どうなればOKなのか」を伝える**

❌ **良くない例：**
```
コードを良くして
```

「良く」ってどういうこと？という疑問が残ります。

✅ **改善した例：**
```
次のPythonコードを改善してください。

今の問題:
- 実行時間: 5秒かかる（1万件のデータを処理）
- メモリ使用量: 500MB使ってる

目指したいゴール:
- 実行時間を2秒以下にしたい
- メモリは200MB以下に抑えたい

特に見てほしいところ:
- 二重ループになっている部分（45〜67行目）
- リストを何度も処理してる部分（89〜102行目）

[ここにコードを貼る]
```

具体的な数字で「今これくらい」「こうしたい」を示すと、AIも改善しやすくなります。

**コツ3: 良い例を見せる**

❌ **良くない例：**
```
エラーメッセージをもっと親切にして
```

✅ **改善した例：**
```
次のエラーメッセージを、ユーザーに優しい表現に変えてください。

今のメッセージ:
"Error: NullPointerException at line 42"

直すときのポイント:
- ユーザーが次に何をすればいいか分かるようにする
- 専門的な言葉（NullPointerExceptionなど）は使わない
- 解決の手順を書く

こんな感じの例:
"ファイルの読み込みに失敗しました。
 次のことを確認してください：
 1. ファイルが本当にあるか
 2. ファイル名は合っているか
 3. ファイルを読む権限があるか"
```

このように「良い例」を見せてあげると、AIは「こういうトーンで書けばいいんだな」と理解できます。

### 2.3 原則その3：構造化（整理して伝えよう）

**ポイント：** 情報をわかりやすく整理して、順序立てて伝えましょう

#### 整理して伝えるコツ

**コツ1: 階層を使って整理する**

見出しや箇条書きを使うと、ぐっと分かりやすくなります：

```
# メインのお願い
データ分析のレポートを作ってください

## ステップ1: データの下準備
- 抜けているデータをどう扱うか決める
- 変な値（外れ値）を見つける

## ステップ2: 分析する
- 平均や中央値などの基本的な数値を出す
- データ同士の関係性を調べる

## ステップ3: レポートにまとめる
- 分かったことを箇条書きにする
- グラフにした方がいい部分を提案する
```

**コツ2: チェックリストを使う**

チェックボックス形式にすると、確認漏れを防げます：

```
次のチェックリストに沿って、コードをレビューしてください：

□ コーディングルール
  □ 変数名や関数名は分かりやすいか
  □ インデントはきれいに揃ってるか

□ ちゃんと動くか
  □ 必要な機能は全部あるか
  □ 特殊なケースでも大丈夫か

□ 速度は問題ないか
  □ ムダな計算をしてないか
  □ 適切なデータ構造を使ってるか

□ セキュリティは大丈夫か
  □ 入力値のチェックはあるか
  □ SQLインジェクション対策はしてるか
```

### 2.4 原則その4：段階性（ステップに分けよう）

**ポイント：** 難しいタスクは小さなステップに分けましょう

#### 段階的に進めるメリット

1. **分かりやすい**: 一つ一つのステップがシンプル
2. **問題を見つけやすい**: どこでつまずいたか分かりやすい
3. **修正しやすい**: 途中で方向転換できる

#### 実例で見てみましょう

❌ **良くない例：**
```
Webアプリを設計して作ってください
```

これだと大きすぎて、どこから手をつければいいか分かりません。

✅ **改善した例：**
```
Webアプリの設計と実装を、段階的に進めてください。

【ステップ1】まず何が必要か整理する
次の機能について、詳しい仕様を箇条書きにしてください：
- ユーザー登録とログイン
- 商品の一覧表示
- カート（買い物かご）
- 支払い処理

（ステップ1の結果を見てから、次に進みます）

【ステップ2】データベースを設計する
ステップ1で決めた機能に必要なテーブルを設計してください。
各テーブルについて：
- テーブル名
- カラム名とデータ型
- 主キーと外部キー
- インデックス

（この後も、同じように段階的に進めます）
```

一気に全部やろうとせず、一つずつ確認しながら進めると失敗が減ります。

### 2.5 原則その5：反復性（何度も試そう）

**ポイント：** 最初から完璧を目指さず、試しながら改善していきましょう

#### 改善のサイクル

```
[最初のプロンプト] → [試してみる] → [結果を見る] → [改善する] → [また試す] → ...
```

#### 実際にやってみた例

**1回目: とりあえず書いてみた**
```
機械学習について説明して
```

**結果：** 範囲が広すぎて、ぼんやりした説明になった

**2回目: もう少し絞ってみた**
```
教師あり学習の分類問題について、初心者向けに説明して
```

**結果：** 良くなったけど、まだちょっとふわっとしてる

**3回目: 構成と例も追加してみた**
```
教師あり学習の分類問題について、プログラミング初心者向けに説明してください。

次の構成でお願いします：
1. 分類問題って何？（100文字くらい）
2. 具体例: スパムメールの判定（200文字くらい）
3. 仕組みの説明（150文字くらい）
4. Pythonでのコード例（10行くらい）
```

**結果：** やっと思い通りの説明が返ってきた！

このように、何度か試しながら調整していくのが普通です。最初から完璧じゃなくて大丈夫ですよ。

### 2.6 原則その6：コンテキスト重視（背景を伝えよう）

**ポイント：** 状況や背景をしっかり伝えると、AIももっと良い答えを返してくれます

#### 背景情報にはこんな種類があります

**種類1: タスク（仕事）の背景**

今どんな状況で、何を解決したいのかを伝えます：

```
# 今の状況
私たちはネットショップを運営しています。
最近、カートに入れたのに買わずに帰る人が60%もいます。
他のお店の平均は40%なので、かなり高いです。

# 何をしたいか
カートに入れた人がちゃんと買ってくれるような改善策を教えてください

# 条件
- 使えるお金: 月50万円まで
- 実現したい期間: 3ヶ月以内
- 使ってる技術: React, Node.js, PostgreSQL
```

**種類2: 誰に向けた内容かの背景**

読む人によって、説明の仕方を変えた方がいいですよね：

```
# 読む人
- 会社の経営陣（プログラミングは分からない人たち）
- 年齢: 50〜60代
- 知りたいこと: 投資対効果、ビジネスへの影響

# 説明の仕方
- 専門用語は使わない
- ビジネスにどう役立つかを中心に
- 数字やグラフで見せる
```

**種類3: 文化や慣習の背景**

国や地域によって、好まれる表現は違います：

```
# どこ向けか
日本市場向けの広告メッセージ

# 気をつけること
- ストレートな言い方より、やんわりした表現が好まれる
- みんなと調和することを大事にする文化
- 品質や信頼を重視する
```

このように背景をしっかり伝えると、AIは「ああ、こういう状況なんだな」と理解して、もっと的確な答えを返してくれます。

## 第3部：テクニックとパターン

ここからは、もっと実践的な「プロンプトの書き方パターン」を学んでいきます。料理のレシピのように、「こういうときはこう書くといいよ」というパターンを覚えておくと便利です。

### 3.1 基本的な3つのパターン

#### 3.1.1 ゼロショット（例なしパターン）

**どんなとき？** 例を見せずに、そのまま「これやって」と頼む方法です

**向いてるタスク：**
- よくあるお願い（翻訳、要約など）
- 説明してほしいとき
- 簡単な質問

**例えば：**
```
次の文を英語に訳してください：
「本日は晴天なり」
```

**良いところ：**
- シンプルで書きやすい
- 文字数が少なくて済む

**気をつけること：**
- 難しいタスクだとうまくいかないことも
- 答えの形式がバラバラになりがち

#### 3.1.2 ワンショット（例1つパターン）

**どんなとき？** 「こんな感じでやってね」と1つ例を見せる方法です

**例えば：**
```
次の例と同じように、文章を要約してください。

例:
入力: 「今日は天気が良かったので、公園に行って散歩しました。桜がとてもきれいでした。」
出力: 「公園散歩、桜鑑賞」

では、こちらも同じようにお願いします：
入力: 「昨日の会議では新製品の開発について話し合いました。予算と締め切りが主な話題でした。」
```

例を1つ見せるだけで、AIは「ああ、こういう短さでまとめればいいんだな」と分かります。

#### 3.1.3 フューショット（例いくつかパターン）

**どんなとき？** 複数の例（だいたい2〜5個）を見せて、パターンをはっきりさせる方法です

**例えば：**
```
次の例を見て、同じように感情分析をしてください。

例1:
テキスト: "この商品は最高です！買って良かった。"
感情: ポジティブ
点数: 0.9

例2:
テキスト: "普通の品質ですね。特に良くも悪くもない。"
感情: 中立
点数: 0.5

例3:
テキスト: "ひどい品質です。もう買いません。"
感情: ネガティブ
点数: 0.1

では、こちらを分析してください：
テキスト: "思ってたよりちょっと良かったです。"
```

**フューショットを効果的に使うコツ：**
1. **いろんなパターンを見せる**: ポジティブ、ネガティブ、中立など、バリエーションを持たせる
2. **何を学んでほしいか明確に**: 例から何を読み取ってほしいかはっきりさせる
3. **形式を揃える**: すべての例で同じフォーマットを使う
4. **数は適度に**: 多すぎると長くなるので、3〜5個くらいがちょうどいい

### 3.2 ちょっと高度なテクニック

基本パターンに慣れてきたら、こんなテクニックも使ってみましょう。

#### 3.2.1 考える過程を見せてもらう（Chain-of-Thought）

**どんなとき？** AIに「どう考えたか」のプロセスを見せてもらう方法です

普通に聞くとこうなります：

```
Q: 太郎くんは15個のりんごを持っています。
   5個を花子さんに、3個を次郎くんにあげました。
   残りは何個ですか？

A: 7個
```

答えは合ってますが、どう計算したか分かりません。

「考える過程を見せて」と頼むとこうなります：

```
Q: 太郎くんは15個のりんごを持っています。
   5個を花子さんに、3個を次郎くんにあげました。
   残りは何個ですか？

A: 順番に考えてみましょう。
1. 最初に持っていたりんご: 15個
2. 花子さんにあげた: 5個
3. 次郎くんにあげた: 3個
4. あげた合計: 5 + 3 = 8個
5. 残り: 15 - 8 = 7個

答え: 7個
```

途中の計算が見えるので、間違いがあったときにどこで間違えたか分かります。

**プログラミングで使うとこんな感じ：**

```
次のコードの処理速度を分析してください。
順番に考えて、各ステップでどれくらい時間がかかるか教えてください。

```python
def find_duplicates(arr):
    duplicates = []
    for i in range(len(arr)):
        for j in range(i+1, len(arr)):
            if arr[i] == arr[j]:
                duplicates.append(arr[i])
    return duplicates
```

こんな順番で分析してください:
1. 外側のループは何回回る？
2. 内側のループは何回回る？
3. 1回のループで何をしてる？
4. 全体でどれくらいの時間がかかる？（O記法で）
```

考える過程を見せてもらうことで、AIの回答の信頼性も上がります。

### 4.3 よくある落とし穴と対策

#### 落とし穴1: 前提知識の不一致

**問題：**
AIがあなたの知識レベルや背景を理解していないため、説明が難しすぎたり簡単すぎたりする。

**症状：**
- 専門用語だらけの説明
- または、逆に当たり前すぎる説明

**対策：**
```
# 学習者プロファイルを明示
私は{{current_level}}です。
{{background_knowledge}}は理解していますが、
{{target_topic}}については初めて学びます。

{{説明してほしい内容}}

説明の際は、以下に注意してください：
- {{already_known}}については説明不要
- {{technical_terms}}などの専門用語は使用時に説明を添える
- 具体例を多用する
```

#### 落とし穴2: 複数タスクの混在

**問題：**
一つのプロンプトに複数の無関係なタスクを詰め込むと、どれも中途半端になる。

**悪い例：**

```

**対策：タスクを分離**
```

```

**問題：**

**悪い例：**

**悪い例：**

## 第5部：最適化と評価

### 5.1 プロンプトの評価方法

#### 5.1.1 評価フレームワーク

**CLEAR評価基準**

| 基準 | 評価項目 | チェックポイント |
|------|---------|----------------|
| **C**larity（明確性） | タスクが明確か | ・指示は一義的か<br>・曖昧な表現はないか<br>・専門用語は定義されているか |
| **L**ength（適切な長さ） | 情報量は適切か | ・過不足はないか<br>・冗長な部分はないか<br>・重要情報は含まれているか |
| **E**xamples（例示） | 例は適切か | ・具体例はあるか<br>・例は多様か<br>・例は正確か |
| **A**lignمent（整合性） | 目的と一致しているか | ・タスクと期待する出力が一致<br>・制約条件が矛盾していない<br>・役割設定が適切 |
| **R**eproducibility（再現性） | 同じ結果が得られるか | ・複数回実行して一貫性はあるか<br>・誰が実行しても同じ解釈か |

#### 5.1.2 定量的評価

**メトリクス例：**

### 5.5 ベストプラクティスまとめ

#### 実践的な10の原則

1. **単純明快に始める**
   - 最初は簡潔なプロンプトから
   - 必要に応じて詳細を追加

2. **反復的に改善**
   - 一度で完璧を求めない
   - フィードバックループを回す

3. **具体例を活用**
   - Few-shot learningの力を使う
   - 多様な例を示す

4. **構造化を心がける**
   - 見出し、箇条書き、番号付きリストを使用
   - 階層的に整理

5. **制約を明示**
   - 曖昧さを排除
   - 測定可能な基準を設定

6. **コンテキストを提供**
   - 背景情報を十分に
   - 対象読者を明確に

7. **出力形式を指定**
   - テンプレートを提示
   - 例を示す

8. **段階的に進める**
   - 複雑なタスクは分解
   - 一つずつ確認

9. **メタ情報を活用**
   - 「考え方を示して」
   - 「ステップバイステップで」

10. **記録と再利用**
    - 成功したプロンプトを保存
    - テンプレート化して再利用

### この記事で学んだこと

長い記事でしたが、お疲れさまでした！ここまで読んでくださって、ありがとうございます。

### これからどうする？

プロンプトを上手に書けるようになるには：

**1. とにかく実践**
   - いろんなことで試してみましょう
   - 失敗しても大丈夫、そこから学べます
   - うまくいったパターンはメモしておきましょう

**2. 他の人から学ぶ**
   - 上手な人のプロンプトを見て勉強
   - 自分の経験を誰かとシェア
   - アドバイスをもらって改善

**3. 少しずつ良くしていく**
   - よく使うプロンプトは保存しておく
   - たまに見直して改善
   - 新しい書き方も試してみる

**4. 使える場面を広げる**
   - いろんな分野で使ってみる
   - 他のAIでも試してみる
   - 自分だけのオリジナルパターンを作る
