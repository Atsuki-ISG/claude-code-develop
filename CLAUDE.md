# 執筆ルール

このドキュメントは、articles配下の記事を執筆する際の文体、構成、表現の指針をまとめたものです。

## 文体の原則

### 1. 基本トーン：親しみやすい「ですます」調

#### ✅ 推奨される表現

**語りかけるような文体**
- 「〜してください」「〜しましょう」で読者を導く
- 「〜んです」「〜なんです」で親近感を出す
- 「〜でしょうか？」で読者に問いかける

```markdown
良い例：
- 「プロンプトって何？なぜ大切なの？」
- 「これが大切なポイントなのですが、**プロンプトの質が、AIの回答の質を決める**んです。」
- 「最初から完璧じゃなくても全然OK。」
```

**励ましと共感の表現**
- 「大丈夫です」「心配ありません」
- 「お疲れさまでした」「ありがとうございます」
- 「少しずつ〜していけば大丈夫です」
- 「〜しても大丈夫、そこから学べます」

```markdown
良い例：
- 「最初から全部完璧にできなくても大丈夫。少しずつ意識していけば、自然と身についていきますよ。」
- 「長い記事でしたが、お疲れさまでした！ここまで読んでくださって、ありがとうございます。」
```

**口語的な表現の活用**
- 「〜ってなんだろう？」「〜って何？」
- 「〜じゃありません」「〜じゃなく」
- 「ぐっと」「ちょっと」「かなり」などの副詞
- 「ふわっとした」「ざっくり」などの柔らかい表現

```markdown
良い例：
- 「プロンプトってなんだろう？」
- 「AIがどう動いているのか、ざっくり理解しておきましょう。」
- 「見出しや箇条書きを使うと、ぐっと分かりやすくなります」
```

#### ❌ 避けるべき表現

**硬い・堅苦しい表現**
```markdown
悪い例：
- 「〜である」「〜だ」などの断定調
- 「〜すべきである」「〜しなければならない」
- 「本稿では」「以下に示す」などの論文調
- 「〜に関して」「〜について述べる」などの形式的な表現
```

**距離を感じさせる表現**
```markdown
悪い例：
- 三人称での説明（「ユーザーは〜」ではなく「あなたは〜」「私たちは〜」）
- 受動態の多用（「〜される」ではなく「〜します」）
```

### 2. 専門用語の扱い方

#### 必ず平易な説明を添える

専門用語を使う場合は、初心者でも分かるように説明を加えます。

```markdown
良い例：
- 「大規模言語モデル（LLMと呼ばれます）」
- 「コンテキストウィンドウ（一度に覚えられる量）」
- 「これを Few-shot Learning と言います」

悪い例：
- 「LLMは〜」（説明なしで略語を使用）
- 「Few-shot Learningを活用する」（専門用語のみ）
```

#### 日本語の平易な言葉を優先

可能な限り、専門用語を日本語で言い換えます。

```markdown
良い例：
- 「背景情報」（コンテキストとも書くが、まず日本語）
- 「やってほしいこと」（タスク）
- 「ルール」（制約条件）

カッコ書きで英語を併記：
- 「役割定義（Role）」
- 「出力形式（Format）」
```

### 3. 文章の長さとリズム

#### 1文は短く、簡潔に

- 1文は2〜3行以内が目安
- 複雑な内容は箇条書きに分解
- 長い説明は段落を分ける

```markdown
良い例：
「プロンプト（Prompt）とは、AI言語モデルに「こうしてほしい」と伝えるためのテキスト入力のことです。私たち人間が、自然な言葉でAIに指示を出すための窓口だと考えてください。」

（2文で簡潔に説明し、比喩を使って理解を助ける）
```

#### 疑問形で興味を引く

セクションの冒頭や小見出しで疑問形を使い、読者の興味を引きます。

**重要：疑問文の語尾は必ず「？」マークをつける**
- 「〜でしょうか？」「〜ませんか？」「〜ですか？」
- 「〜でしょうか。」「〜ませんか。」は ❌

```markdown
良い例：
- 「なぜプロンプトが重要なの？」
- 「トークンって何？AIはどうやって文章を作るの？」
- 「これだと何が問題でしょうか？」
- 「AIに「コードを書いて」とだけ伝えて、全く期待と違うコードが返ってきた経験はありませんか？」

悪い例：
- 「なぜプロンプトが重要なの。」（疑問文なのに「。」）
- 「経験はありませんか。」（疑問文なのに「。」）
```

## 構成の原則

### 1. 明確な階層構造

#### 大きな区切りは「第〇部」

長い記事は第1部、第2部のように大きく区切ります。

#### 各セクションに「なぜその話をするのか」のリード文を追加

各セクション（特に大きなセクション）の冒頭には、以下を含むリード文を追加します：

1. **なぜその話をするのか**：読者がそのセクションを読む動機を与える
2. **体験談・考察（適度に）**：筆者の経験や気づきを交えて親近感を出す
3. **次の展開の予告**：このセクションで何を学ぶかを明示

```markdown
良い例：

## 第2部：プロンプト設計の原則

第1部で基本を学んだので、ここからは「実際にどう書けばいいの？」という実践的な話に入ります。正直に言うと、私も最初はこの原則を知らずに、試行錯誤ばかりしていました。でも、この6つの原則を意識するようになってから、一発で期待通りの結果が返ってくる確率がぐっと上がったんです。

ここからは、「良いプロンプト」を書くための6つの大切な考え方を学んでいきます。
```

**リード文のバランス：**
- すべてのセクションに体験談を入れる必要はない（かえって冗長になる）
- 読者がイメージしやすい場合、または重要な転換点で使う
- 自然な流れを保つことを優先する

```markdown
# 記事タイトル：サブタイトル

## 目次
1. [第1部：〇〇](#第1部〇〇)
2. [第2部：〇〇](#第2部〇〇)

---

## 第1部：〇〇

### 1.1 セクション名

#### サブセクション名
```

#### 各セクションは「ポイント」「コツ」で要点明示

読者が何を学ぶかを最初に明示します。

```markdown
### 2.1 原則その1：明確性（はっきり伝えよう）

**ポイント：** あいまいな表現を避けて、何をしてほしいのか正確に伝えましょう

#### 明確に伝えるコツ

**コツ1: 5W1Hを意識する**
**コツ2: あいまいな言葉に注意**
**コツ3: 良い例と悪い例を比べてみよう**
```

### 2. 具体例を豊富に使う

#### 良い例と悪い例の対比

❌と✅を使って視覚的に分かりやすく。

```markdown
❌ **良くない例：**
```
（悪い例のコード）
```

これだと何が問題でしょうか？
- 理由1
- 理由2

✅ **改善した例：**
```
（良い例のコード）
```

こうすると、AIは「〜」と理解できます。
```

#### 実際のデータやコードを示す

抽象的な説明だけでなく、必ず具体例を添えます。

```markdown
良い例：
**例えば：**
```
次の売上データを分析して、傾向とおかしな値を見つけてください：

2024年1月: 150万円
2024年2月: 148万円
...
```
```

### 3. 視覚的な整理

#### 箇条書きと番号付きリストを活用

情報は箇条書きや番号付きリストで整理します。

```markdown
**伝えるべき情報：**
- 誰に向けた内容なのか（技術レベル、年齢、経験）
- どんな環境で使うのか
- 今どんな問題や状況があるのか

**コツ：**
- 「〜してください」という動詞で始めましょう
- あいまいな表現を避けましょう
- 1つのプロンプトには1つのメインタスクを
```

#### 表で情報を比較

対比や比較は表形式で見やすく。

```markdown
| あいまいな表現 | もっと明確な表現 |
|----------|----------|
| "詳しく説明して" | "500文字くらいで、原因・影響・対策の3つを説明して" |
| "良いコードを書いて" | "読みやすくて、処理速度がO(n)以下になるコードを書いて" |
```

#### コードブロックで例示

コードやプロンプトの例は必ずコードブロックで囲みます。

````markdown
```python
def example():
    pass
```

```
プロンプトの例をここに書く
```
````

### 4. まとめセクション

#### 各パートの最後に要点を復習

読者が学んだことを整理できるように、まとめを入れます。

```markdown
## まとめ

### この記事で学んだこと

**第1部：基礎知識**
- ポイント1
- ポイント2

**第2部：〇〇**
- ポイント1
- ポイント2

### これからどうする？

**1. とにかく実践**
   - アクション1
   - アクション2

**2. 他の人から学ぶ**
   - アクション1
   - アクション2
```

#### 励ましの言葉で締めくくる

記事の最後は、読者を励まし、次のアクションを促します。

```markdown
### 最後に一言

〇〇は、ただのテクニックじゃありません。「〜」というスキルです。

このスキルは、練習すれば必ず上達します。最初から完璧じゃなくても全然OK。試行錯誤を楽しみながら、少しずつ上手になっていってください。

この記事が、あなたの〇〇の旅の役に立てたらうれしいです。さあ、早速〜してみましょう！
```

## 表現の指針

### 1. 読者との距離を縮める表現

#### 体験談と考察で親近感を出す

適度に筆者の体験や気づきを混ぜることで、読者との距離が縮まります。

**使いどころ：**
- 新しいセクションのリード文
- 重要なポイントを強調したいとき
- よくある失敗や落とし穴を説明するとき
- 読者が「本当に効果あるの？」と疑問に思いそうなとき

```markdown
良い例：
- 「私がこのテクニックを知ったのは、複雑なアルゴリズムの説明を求めた時でした。最初は答えだけもらっていたのですが、「本当に合ってるのかな？」と不安だったんです。」
- 「私も最初は「一発で完璧なプロンプトを書かないと」と思っていましたが、実際には3〜4回の試行錯誤が当たり前だと気づきました。」
- 「特に落とし穴2（複数タスクの混在）は、締め切り間際に焦って「全部一気にやってもらおう」と思って失敗した苦い思い出があります（笑）。」
```

**注意点：**
- すべてのセクションに体験談を入れる必要はない
- 体験談が主役にならないよう、簡潔に（2〜3文程度）
- 「私は〜」が連続しないよう配慮する
- 体験談→本題への自然な流れを作る

#### 二人称で語りかける

「ユーザー」ではなく「あなた」「私たち」を使います。

```markdown
良い例：
- 「あなたが入力したテキストを読んで」
- 「私たち人間が、自然な言葉で」
- 「一緒に学んでいきましょう」

悪い例：
- 「ユーザーが入力したテキストを」
- 「利用者は〜」
```

#### 共感と寄り添い

読者の気持ちに寄り添う表現を使います。

```markdown
良い例：
- 「これだと何が問題でしょうか？」（読者と一緒に考える）
- 「データがどんなものか分からないと、AIも困ってしまいます。」（共感）
- 「「良く」ってどういうこと？という疑問が残ります。」（読者の疑問を代弁）
```

### 2. 段階的な説明

#### 「まず〜」「次に〜」「最後に〜」で順序を明示

```markdown
**1. トークン化**
AIは、あなたが入力した文章を「トークン」という小さな単位に切り分けて読んでいます。

**2. 確率で次の言葉を予測**
AIは、学習したデータのパターンから「次に来る言葉は何が一番自然か？」を確率で計算して選んでいます。

**3. コンテキストウィンドウ（一度に覚えられる量）**
AIが一度に読める文章の量には限界があります。
```

#### 実例→説明の流れ

抽象的な説明の前に、具体例を示します。

```markdown
良い例の流れ：
1. 「例えば：」で具体例を提示
2. 「このように〜」で説明
3. 「つまり、〜ということです」でまとめ
```

### 3. 強調と注目

#### 太字で重要ポイントを強調

```markdown
そして、ここが大切なポイントなのですが、**プロンプトの質が、AIの回答の質を決める**んです。
```

#### 絵文字は使わない

親しみやすさは文体で表現し、絵文字は使用しません。

```markdown
❌ 悪い例：
「頑張りましょう！💪」

✅ 良い例：
「少しずつ上手になっていってください。」
```

### 4. 否定的な表現を避ける

#### 「できない」より「〜する方法」

```markdown
悪い例：
「これではうまくいきません」

良い例：
「もっと良い方法があります」
「こう改善するともっと良くなります」
```

#### 「難しい」より「一緒に学ぼう」

```markdown
悪い例：
「これは難しい概念です」

良い例：
「少し複雑ですが、一緒に見ていきましょう」
「ざっくり理解しておきましょう」
```

## 具体的な執筆プロセス

### 1. 構成を先に決める

1. 大まかな目次を作成（第〇部：〇〇）
2. 各セクションの見出しを決める
3. 各セクションで伝えたいポイントを箇条書き
4. 具体例を考える
5. 本文を執筆

### 2. 執筆時のチェックリスト

#### 文体チェック
- [ ] 「ですます」調で統一されているか
- [ ] 読者に語りかける表現を使っているか
- [ ] 専門用語に説明を添えているか
- [ ] 1文が短く、簡潔か

#### 構成チェック
- [ ] 見出しの階層は適切か
- [ ] 各セクションに「ポイント」や「コツ」が明示されているか
- [ ] 良い例と悪い例の対比があるか
- [ ] 具体例は豊富か
- [ ] まとめセクションがあるか

#### 表現チェック
- [ ] 励ましや共感の表現があるか
- [ ] 疑問形で興味を引いているか
- [ ] 箇条書きや表で整理されているか
- [ ] 重要なポイントは太字で強調されているか
- [ ] 否定的な表現を避けているか

### 3. 執筆後の見直し

1. 音読してリズムを確認
2. 読者目線で「分かりやすいか」をチェック
3. 専門用語に説明があるか確認
4. 具体例が適切か確認
5. まとめで全体を復習できるか確認

## フィードバック反映のプロセス

記事に対してフィードバックを受けた際は、以下のプロセスで対応します：

### 1. フィードバックの記録と分析

受け取ったフィードバックを具体的に記録します：
- どのような指摘か（文体、構成、表現など）
- 具体例があれば記録する
- 影響範囲を確認する（一部 or 全体）

### 2. 記事への反映

フィードバックに基づいて記事を修正します：
- 該当箇所を特定して修正
- 同様の問題が他にないか確認
- 修正後の整合性をチェック

### 3. 抽象化と学びの抽出

個別のフィードバックから、一般化できる原則を抽出します：

**例：**
```
【フィードバック】
「疑問文の語尾に「。」ではなく「？」マークをつけてください」

【抽象化した学び】
- 疑問文は必ず「？」で終わる
- 読者への問いかけは明確に表現する
- 文末記号は文の種類（平叙文・疑問文・感嘆文）を明確にする役割がある

【執筆ルールへの反映】
→「疑問形で興味を引く」セクションに「疑問文の語尾は必ず「？」マークをつける」という注意事項を追加
```

### 4. CLAUDE.mdへの反映

抽象化した学びを適切なセクションに追加します：
- 既存のルールの拡充 or 新規セクションの追加
- 具体例を必ず含める（良い例・悪い例）
- なぜそのルールが重要かの理由も記載

### 5. 更新履歴への記録

CLAUDE.mdの「更新履歴」セクションに以下を記録します：
- 日付
- 追加・修正した内容の概要
- フィードバック元（記事名など）

**テンプレート：**
```markdown
- YYYY-MM-DD: [フィードバック内容]を反映（対象記事: XXX）
  - 追加: [新しく追加したルール]
  - 修正: [既存ルールの修正内容]
```
